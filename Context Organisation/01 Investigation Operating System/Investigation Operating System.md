# Investigation Operating System

## Definition

The investigation operating system is the method by which research becomes organized, judged, and reused.

It is not a single workflow tool.
It is the combination of routines, decomposition rules, model routing, synthesis practices, and archive discipline that turns scattered inquiry into a persistent system.

## Why It Matters

The original archive shows that the strongest recurring pattern is not tool collecting.
It is method design.

Again and again, the archive returns to questions such as:

- How should a task be decomposed?
- Which model should handle which stage?
- When does a prompt become an agent?
- How should long research sessions be compacted into reusable knowledge?
- How do we prevent useful work from dissolving into clutter?

Those questions define the real center of gravity.

## Durable Principles

### Research should end in structure

A useful investigation produces one of three outputs:

- a decision
- a reusable note
- an operational next step

If it produces none of these, it probably should not survive.

### Expensive reasoning should be reserved for high-value steps

Not every stage deserves the best model or the longest context.
The archive repeatedly points toward routing by task type rather than treating one model as the universal brain.

### Notes should compress, not merely accumulate

The old archive contains many examples of repeated subjects spread across many notes.
The new system prefers fewer notes with sharper summaries and clearer implications.

### The system must be legible to future operators

A note is more valuable when it explains:

- what the subject is
- why it matters
- what the current stance is
- what should happen next

## Operating Test

A method deserves preservation when it helps answer one of these:

- What should be studied next?
- What should be ignored?
- What should be delegated to a tool or agent?
- What should be turned into a project?
- What should be turned into a durable concept note?
