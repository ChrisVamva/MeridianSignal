## Section 1 — Handoff Audit

Every transition between agents examined for contract integrity.

---

### Handoff 1: Brainstormer → Evaluator

**What passes:** ideas.md — structured ideas with 8 fields each, plus the brainstormer's own Top 3 and winner recommendation.

**Contract status:** Mostly solid. The Evaluator's input contract matches the Brainstormer's 7-field output structure.

**Gap 1 — Anchoring contamination.** The brainstormer produces its own Top 3 and winner recommendation. The Evaluator receives this ranking embedded in the document. There is no instruction telling the Evaluator to ignore it. The brainstormer's ranking will anchor the Evaluator's judgment — the Evaluator will be unconsciously biased toward confirming the brainstormer's winner rather than evaluating independently.

**Fix:**

```
Add to Evaluator prompt, before the scoring section:

"The input document may contain a ranking or winner 
recommendation produced by the brainstormer. 
Ignore it entirely. 
Do not let the brainstormer's ranking influence your 
scoring in any direction — positive or negative.
Your evaluation is independent. Start from the 
raw idea fields only."
```

**Gap 2 — No document naming convention.** The brainstormer produces output but neither prompt specifies what the output file should be called or how it should be structured for handoff. Two pipeline runs produce documents with no way to trace which belongs to which run.

**Fix (system-level — addressed in Section 3).**

---

### Handoff 2: Evaluator → Research Agent

**What passes:** The winning idea's full 7-field entry plus the Evaluator's verdict on it.

**Gap — "Full idea report" is undefined.** The Research Agent prompt says _"[PASTE THE FULL IDEA REPORT HERE]"_ — but after the Evaluator runs, there are two possible interpretations of "full idea report":

- The original 7-field brainstormer entry for the winning idea only
- The Evaluator's complete output including scores, rejected ideas, and Top 3

The Research Agent only needs the winner's fields plus the evaluator's verdict on it. But it receives no instruction on what to extract. In practice, users paste the entire evaluation.md — the Research Agent then reads rejected ideas and may be influenced by them.

**Fix:**

```
Add to Research Agent input section:

"Winner Idea Report format expected:
- Idea Name
- Core Thesis
- Problem
- Solution  
- Why Now
- Strategic Edge
- Zero-Cost Lever
- Evaluator verdict (confidence level + main risk 
  from the Top 3 section)

Do not paste the full evaluation including rejected ideas.
Rejected ideas are not relevant to this research task
and will contaminate the analysis."
```

---

### Handoff 3: Research Agent → Variation Agent

**What passes:** research_brief.md + winner_idea_report.md

**Contract status:** Well-specified. The Variation Agent's input mapping is explicit about which sections of the research brief drive which variation dimensions.

**Gap — Hidden Opportunity (Section 9) treatment.** The Variation Agent is told to treat the research brief's Hidden Opportunity as _"a variation seed, not a conclusion."_ This is correct. But the Research Agent has no corresponding instruction — it writes Section 9 as a conclusion. The Variation Agent may receive a Section 9 that is already fully formed as a product recommendation, which creates pressure to treat it as a pre-decided variation rather than raw material.

**Fix:**

```
Add to Research Agent Section 9 instruction:

"Write Hidden Opportunity as a raw signal, not a 
product recommendation. 
Format: 'The research surfaced [signal]. This could 
suggest [direction] — but product form is not decided here.'
The Variation Agent will develop this signal. 
Do not resolve it into a product concept."
```

---

### Handoff 4: Variation Agent → Synthesis Agent

**What passes:** variations.md + research_brief.md + winner_idea_report.md

**Contract status:** The strongest handoff in the pipeline. The Load-Bearing Quality field was specifically designed as a forward contract for the Synthesis Agent. The Note to Synthesis Agent in the variations.md header calibrates it immediately.

**Gap — No version or run identifier in variations.md header.** The Synthesis Agent receives variations.md but has no way to confirm it was produced from the same research brief it also receives. If a user mixes documents from different pipeline runs (which will happen), the Synthesis Agent has no detection mechanism.

**Fix (system-level — addressed in Section 3).**

**One internal gap:** The Synthesis Agent's Step 2 (Compatibility Assessment) asks about conflicts between qualities but gives no instruction for what to do when ALL qualities from one variation conflict with the core. That variation becomes invisible — none of its qualities survive — but the Variation Agent explicitly prohibited discarding variations. The Synthesis Agent should acknowledge when a full variation contributed nothing.

**Fix:**

```
Add to Synthesis Agent Step 3 (Core Identity Selection):

"If all qualities from a specific variation were cut,
note it explicitly:
'Variation [X]: no quality survived to the final set.
Reason: [all qualities conflicted with / were 
enhancements relative to the surviving core].'
This is a valid outcome. Do not force a quality from 
every variation into the synthesis."
```

---

### Handoff 5: Synthesis Agent → Design Agent ⚠️ CRITICAL GAP

**What passes:** product_concept.md + research_brief.md

**Contract status:** BROKEN. This is the most significant gap in the entire pipeline.

The Design Agent was written and revised before the Variation and Synthesis Agents were added. Its current input specification says:

_"You are receiving a validated market and production research brief."_

That's it. The Design Agent has no instruction to receive, read, or anchor on product_concept.md. When you run the pipeline, you will paste both documents — but the Design Agent will treat the research brief as its primary input and may treat product_concept.md as supplementary context or ignore it entirely.

The entire purpose of the Variation and Synthesis agents was to give the Design Agent a richer, more specific product definition. That purpose fails if the Design Agent doesn't know product_concept.md is its primary anchor.

**Fix — Design Agent input section must be updated:**

```
### Input

**Primary Input — Synthesized Product Concept:**
[PASTE FULL product_concept.md HERE]

This is your primary design anchor.
Section 9 (Design Brief Handoff) of product_concept.md 
is your direct briefing. Read it first.
The 2-3 load-bearing qualities in Section 2 are 
non-negotiable — every design decision must serve them.
The scope boundary in Section 9 defines what you 
must not build.
The one quality marked as never-to-be-compromised 
cannot be traded away in any design decision.

**Supporting Input — Validated Research Brief:**
[PASTE FULL research_brief.md HERE]

Use the research brief for:
- Architecture decisions → Production Path (Section 6)
- External dependencies → Open-Source Leverage (Section 7)
- Constraints → Business Risks (Section 8)
- User model details → User Segments (Section 4)

Do not derive the product shape from the research brief.
The product shape is defined in product_concept.md.
The research brief informs how to build it,
not what to build.
```

---

### Handoff 6: Design Agent → Planner

**What passes:** design.md + research_brief.md

**Contract status:** Solid. The Planner's input mapping explicitly references design.md sections and research brief sections.

**Gap — product_concept.md is not passed to the Planner.** The Planner receives design.md and research_brief.md — but not product_concept.md. This means the Planner has no access to the Superiority Argument (Section 7) or the explicit discards (Section 3) from the synthesis. These matter for planning: the discards define scope boundaries that should prevent scope creep during execution, and the Planner should know them.

**Fix:**

```
Add to Planner input section:

**Scope Reference — Synthesized Product Concept:**
[PASTE product_concept.md Section 3 (Explicit Discards) 
and Section 9 (Design Brief Handoff) ONLY]

Use this for:
- Scope boundary enforcement: tasks that implement 
  discarded qualities are out of scope — flag them
- The one never-to-be-compromised quality: 
  no task may trade this away for speed or simplicity
  
Do not use this to re-derive the product shape.
design.md is the architecture authority.
```

---

### Handoff 7: Planner → Executor

**What passes:** plan.md + design.md + current task ID + prior task reports

**Contract status:** Strong. The Executor's input contract is explicit and the continuation protocol (prior reports required) is well-enforced.

**Gap — product_concept.md scope boundary not passed to Executor.** Same issue as Handoff 6, one step further. The Executor has no access to the explicit discards from synthesis. When executing, the Executor may implement something that was explicitly discarded during synthesis — scope creep that entered through the plan undetected.

**Fix:**

```
Add to Executor input section, after design.md:

**Scope Boundary Reference:**
[PASTE product_concept.md Section 3 (Explicit Discards) ONLY]

If any task in plan.md appears to implement a 
discarded quality, stop immediately.
Report: ESCALATION NEEDED — Task [ID] may implement 
a discarded quality: [name it].
Do not execute the task until the conflict is resolved.
```

## Section 2 — System-Level Issues

Problems that don't belong to any single agent but affect the whole pipeline.

---

### System Issue 1: No Document Registry

Your pipeline produces 8 documents: `ideas.md` → `evaluation.md` → `winner_idea_report.md` → `research_brief.md` → `variations.md` → `product_concept.md` → `design.md` → `plan.md`

There is no system-level document that tracks:

- Which versions of each document belong to the same pipeline run
- When each was produced
- Which agent produced it from which inputs

When you run the pipeline for a second product, you will have two sets of these documents with no naming convention. You will mix them. This is guaranteed.

**Fix — Add a Pipeline Run Header to every document:**

```
Every agent output must begin with:

---
PIPELINE RUN: [product name] — [run date]
Document: [document name]
Produced by: [agent name]
Input documents: [list with dates]
Version: [1.0 / 1.1 if revised]
---
```

This costs nothing and makes the entire pipeline auditable.

---

### System Issue 2: The Zero-Cost Constraint Is Defined Nowhere Centrally

The constraint appears in 7 of 8 prompts — but as a placeholder each time: _"[paste your stated budget here]"_ or _"[your stated constraint]."_ It is never defined in one place and propagated. If you change your budget from $0 to $20/month, you update 7 prompts.

**Fix — Create a Pipeline Constants block:**

```
PIPELINE CONSTANTS
(paste this at the top of every agent prompt)

Zero-cost constraint: [your number — e.g. $0/month]
Team size: [e.g. solo operator]
Target first user segment: [derived from research brief 
                            and carried forward]
2026 context signals: [your 3-5 signals]
Banned product language: platform, solution, ecosystem, 
  seamless, powerful, robust, leverage, innovative

These constants override any agent's default assumptions.
```

One block. Paste it everywhere. Change it once, update everywhere.

---

### System Issue 3: No Upstream Escalation Path

Your Executor has a clear escalation protocol — but it only escalates to you. When the Executor finds a flaw in the plan, you receive a report. Then what? There is no defined protocol for sending a finding _back up the pipeline_ to the appropriate agent.

Examples:

- Executor finds the plan has a task that contradicts design.md → needs Planner
- Planner finds the design has an unresolvable ambiguity → needs Design Agent
- Design Agent finds product_concept.md contradicts research_brief.md → needs Synthesis Agent

Without an upstream escalation path, every issue lands on you as a manual re-entry problem. You become the router for all inter-agent conflicts.

**Fix — Add an Escalation Routing Table to the Pipeline Constants:**

```
ESCALATION ROUTING

Task-level blocker → Report to human. Human issues GO.
Plan-task contradiction → Re-run Planner with flagged section.
Plan-design contradiction → Re-run Design Agent 
  with Open Question added.
Design-concept contradiction → Re-run Synthesis Agent 
  with conflict noted in Open Questions.
Concept-research contradiction → Re-run Research Agent 
  with targeted question.
Research-idea contradiction → Re-run Evaluator 
  with new evidence.

Re-runs are targeted — feed only the affected section 
back to the relevant agent, not the full pipeline.
```

This doesn't automate the routing — but it gives you a decision framework so you're not improvising under pressure when something breaks mid-execution.

---

### System Issue 4: The Prompt Diagnosis Tool Has No Trigger

You built a Prompt Diagnosis tool (separated from the Evaluator prompt). It exists but has no defined trigger condition in the pipeline. It currently lives as a standalone file with no instruction for when to invoke it.

**Fix — Define the trigger:**

```
Invoke prompt_diagnosis.md when:
- The Evaluator rejects more than 60% of brainstorm ideas
- The Research Agent returns "Insufficient evidence" 
  on more than 3 sections
- The Variation Agent produces fewer than 5 variations 
  that pass the differentiation test
- The Synthesis Agent cannot complete the Superiority 
  Argument with specific claims

In each case: feed the weak output + the prompt that 
produced it to prompt_diagnosis.md before re-running.
```

---

### System Issue 5: Context Window Accumulation

By the time the Design Agent runs, it must hold in context:

- product_concept.md (~2,000 words)
- research_brief.md (~2,000 words)

By the time the Planner runs:

- design.md (~3,000+ words)
- research_brief.md (~2,000 words)
- product_concept.md sections (~500 words)

By the time the Executor runs for late-phase tasks:

- plan.md (~4,000+ words)
- design.md (~3,000+ words)
- All prior task reports (accumulating)

This is a real operational risk. Late-phase executor tasks may exceed context windows for some models, causing silent truncation of prior task reports — which means the Executor loses memory of what was completed.

**Fix:**

```
Add to Executor prompt:

Context management:
If prior task reports exceed [X] tokens, compress them:
For each completed task, retain only:
- Task ID
- Status (COMPLETE / DEVIATED)
- Output produced (one sentence)
- Any unresolved deviations

Full reports are archived but not carried forward 
after compression.
Compression threshold: [set based on your model's 
context window]
```

---

### System Issue 6: No Re-entry Points Defined

Your pipeline is linear — it runs top to bottom. But real product development is not linear. You will need to:

- Run the pipeline again for a different idea (full re-run)
- Update the design after execution reveals a problem (partial re-run from Design Agent)
- Validate a new variation without redoing the brainstorm (entry at Variation Agent)

There is no defined re-entry protocol. Currently, starting mid-pipeline means manually assembling inputs with no guidance on what's needed.

**Fix — Define re-entry points:**

```
PIPELINE RE-ENTRY POINTS

Full re-run: start at Brainstormer with new constants.

New idea, same research: start at Variation Agent.
Inputs needed: research_brief.md + new winner_idea_report.md

Design revision only: start at Design Agent.
Inputs needed: product_concept.md + updated research_brief.md
Trigger: execution revealed a structural problem

Plan revision only: start at Planner.
Inputs needed: updated design.md + research_brief.md
Trigger: design changed, plan is now misaligned

Execution restart (same plan): start at Executor.
Inputs needed: plan.md + design.md + all prior task reports
Trigger: execution interrupted, resuming
```

## Section 3 — Agent-Level Residual Gaps

Issues within individual agents not caught in previous evaluations.

---

### Brainstormer — One Residual Gap

The revised brainstormer has a _"[Add your own 2026 signals here]"_ placeholder in the context block. This is intentional — you fill it. But there is no instruction on how many signals to add or what makes a signal useful vs. noise. Users will either add too many (diluting the focus) or too few (leaving the context block nearly empty).

**Fix:**

```
Add 3–5 signals. Each signal must be:
- Specific (not "AI is growing")
- Observable today (not a prediction)
- Relevant to the problem space you are exploring
Example of weak signal: "AI adoption is increasing."
Example of strong signal: "Cursor and Copilot have made 
  solo developers 3x more productive at code generation 
  but zero times more productive at system design."
```

---

### Research Agent — One Residual Gap

The inline citation requirement (_[Source: name/URL]_) is correct — but there is no instruction for what to do when a real source cannot be found for a claim the agent believes to be true. The agent will either hallucinate a source or omit the citation silently. Neither is acceptable.

**Fix:**

```
Add to Research Agent constraints:

If a claim is believed true but no source can be 
cited directly, mark it:
[INFERENCE - no direct source: reasoning is X]

Never fabricate a source URL or publication name.
A believable-sounding fake citation is worse than 
an acknowledged gap.
```

---

### Evaluator — One Residual Gap

The Evaluator's confidence levels (High / Medium / Low) were defined in the revision. But the Evaluator produces a confidence level for the Top 3 — not for the final winner specifically. The Research Agent, Variation Agent, and Design Agent all receive the winner's idea report but never see the confidence level the Evaluator assigned to it.

**Fix:**

```
Add to Evaluator Final Winner section:

Include the confidence level in the winner's 
idea report extract:
"Winner Confidence: [High/Medium/Low]
Reason: [one sentence]
Main unverified assumption: [one sentence]"

This travels with the winner_idea_report.md 
through the entire pipeline.
Every downstream agent knows how validated 
this winner actually is.
```

---

### Design Agent — One Residual Gap (Post-Integration)

Now that the Design Agent receives product_concept.md as primary input, its 8-section schema needs one addition. Currently Section 7 (Constraints and Non-Negotiables) has no field for _"qualities that must never be compromised."_ This came from the Synthesis Agent's Section 9 — it's a direct handoff instruction — but the Design Agent's schema doesn't have a home for it.

**Fix:**

```
Add to Design Agent Section 7:

Non-negotiable quality (from product_concept.md):
[name the quality]
Design constraint: no decision in this document 
may reduce or compromise this quality.
If a design tradeoff would affect it, flag it 
in Open Questions — do not resolve it silently.
```

## Section 4 — What Is Genuinely Solid

A pipeline audit is not useful if it only finds problems. Here is what is working well and should not be changed.

|Element|Why It Works|
|---|---|
