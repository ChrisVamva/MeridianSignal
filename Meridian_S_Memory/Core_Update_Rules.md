# Core Update Rules

## Purpose

These rules define how `Agent_S_Memory` should be updated.

They exist to prevent the system from collapsing into a second archive.

---

## Rule 1: Update By Delta, Not By Volume

New material enters the memory system only when it changes one of these:

- orientation
- continuity
- project understanding
- frontier logic
- durable terrain

If a note adds detail but does not change any of those, it stays in source storage.

---

## Rule 2: Compress Before You Promote

Raw research does not move directly into memory.

First ask:

- what is the core claim?
- what changed?
- why does it matter?
- where does it belong?

Only then can the material be promoted into `Agent_S_Memory`.

---

## Rule 3: Source Folders Are Not Memory

Folders like:

- `Workplace`
- project report collections
- tool landscapes
- experiments

are source reservoirs.

They feed memory.
They are not memory themselves.

---

## Rule 4: Stable Files Resist Churn

Stable files should only be changed when the update genuinely alters the long-term map.

Examples:

- research identity sketch
- research terrain map
- project lines
- core research compressions

Do not rewrite these because of temporary excitement.

---

## Rule 5: Living Files Absorb Current Change

Living files are where new pressure first appears.

Examples:

- active investigations
- open questions
- idea frontier
- repetition boundaries

When a new batch arrives, test it here first before promoting it into stable memory.

---

## Rule 6: Project Memory Must Preserve Function, Not Chatter

A project update belongs in memory only if it clarifies:

- what the project really is
- what line of ambition it belongs to
- what durable lesson it teaches
- how it changes future project thinking

Implementation chatter stays outside the memory core unless it reveals a reusable pattern.

---

## Rule 7: Tool Notes Need Architectural Consequence

A tool update belongs in memory only if it changes:

- model routing logic
- observability strategy
- execution method
- replacement judgment
- system design

Feature lists and product snapshots are not enough.

---

## Rule 8: Frontier Files Track Tension, Not Hype

An idea enters the frontier only if it creates a real next question, such as:

- a new unresolved tension
- a genuine synthesis opportunity
- a practical mechanism not previously present
- a meaningful extension of a project line

Interesting topics alone do not qualify.

---

## Rule 9: Repetition Checks Are Mandatory

Before adding or revising memory, ask:

1. Is this genuinely new to the system?
2. Does this deepen an existing theme or merely rename it?
3. Does this belong to terrain, projects, active investigations, or frontier?
4. Would an agent be more useful after this update?

If the answer to the last question is no, the update should not enter memory.

---

## Rule 10: Every Update Must Name Its Layer

Every accepted update should be classed as one of:

- terrain update
- project memory update
- active investigations update
- frontier update
- compression update
- protocol update

This prevents random note growth.

---

## Rule 11: Memory Files Must Stay Agent-Legible

Any update to `Agent_S_Memory` should preserve:

- direct language
- explicit purpose
- clear distinctions
- low rhetorical noise

The memory system should be readable fast by both humans and agents.

---

## Rule 12: Human Sovereignty Stays In Force

At this stage:

- humans approve structural changes
- agents can assist synthesis
- agents do not autonomously redefine the memory architecture

This rule stands until the research identity and protocol layer are mature enough for stronger automation.

---

## Final Standard

The standard for every update is simple:

Does this make `Agent_S_Memory` more useful as a thinking substrate?

If yes, compress it and place it carefully.
If no, preserve it in the source layer and move on.
