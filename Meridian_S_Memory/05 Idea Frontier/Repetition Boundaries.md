# Repetition Boundaries

## Purpose

This note exists to help agents avoid repeating well-covered territory.

## Already Strongly Explored Zones

### Generic tool collection

The archive already contains a large amount of tool scanning and comparison.
New work should add a sharper verdict, a better synthesis, or a deeper architecture implication.

### Broad claims about agents

The system already knows that agents matter, that loops matter, and that tool use changes the game.
Useful new work must go beyond generic affirmation.

### Planner / executor split as slogan

The system now already understands the broad value of planning models and bounded execution models.
Future work must add a real mechanism, protocol, or implementation consequence.

### Archive organization for its own sake

The system has already moved through archive cleanup and curated organization.
New organization should only happen when it creates a genuinely better research or memory function.

### Sovereignty as slogan

Local-first and sovereign systems are already major themes.
Future work should clarify tradeoffs, implementation pathways, or novel uses rather than repeating the preference.

### Tool stack accumulation

Routing, telemetry, model registries, and agent frameworks are now heavily represented.
New additions should explain architectural consequence, not simply extend the tool catalog.

## Good Anti-Repetition Questions

- What does this add that the system does not already know?
- Does this resolve an open tension or just restate a theme?
- Is this a new mechanism, or just a new label?
- Which existing project line does this extend?
