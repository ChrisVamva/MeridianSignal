# Agent_S_Memory Design

## Mission

`Agent_S_Memory` is the next layer after `Code Archive 2` and `LUX`.

It is not:

- a replacement for the original archives
- a complete agent constitution
- a final doctrine of identity
- an autonomous memory system that rewrites itself

At this stage, it is a prototype memory environment.

Its purpose is to give one or more agents a highly structured line of sight into your research world without prematurely turning your evolving thinking into rigid law.

---

## Why This Exists

`Code Archive 2` proved that your material can be organized as a durable research system.
`LUX` proved that compressed, agent-facing context can be useful in practice.

Each one solves a different problem:

- `Code Archive 2` organizes long-term research knowledge.
- `LUX` compresses active context for an operating agent.

The new product should surpass both by becoming:

an intentionally layered memory environment for agent-assisted research.

This means the new system should help agents do three things well:

1. understand what your research is actually about
2. see what is active, unfinished, or newly emerging
3. propose useful next moves without repeating settled territory

---

## Core Design Constraint

You are still researching and still finding your own rhythm.

That means this system must not behave like a final scripture.
It must be structured enough to help agents, but open enough to let your worldview continue forming.

So the first version should optimize for:

- clarity
- layered access
- non-redundancy
- active orientation
- idea generation from context

It should not yet optimize for:

- hard-coded behavioral commandments
- autonomous self-updating memory
- strongly enforced voice or philosophy rules
- irreversible agent constitutions

---

## What New Product Emerges From Code Archive 2 + LUX

The new product is not simply a merge.

It is a memory operating layer with three interacting dimensions:

### 1. Research Substrate

This comes primarily from `Code Archive 2`.

It provides:

- durable subjects
- conceptual organization
- distilled project lessons
- strategic filters

### 2. Agent-Facing Compression

This comes primarily from `LUX`.

It provides:

- direct orientation
- active context
- compressed summaries
- immediate relevance

### 3. Frontier Logic

This is the new layer that neither predecessor fully solved.

It should provide:

- what has already been explored
- what remains unresolved
- where promising gaps exist
- how to propose non-repetitive next ideas

This third dimension is what makes the new system worthy of existing.

---

## Guiding Principles

### 1. Layer Memory By Function

Not every memory file should do the same job.

Some files should orient.
Some should compress.
Some should track what is active.
Some should mark the frontier.

### 2. Separate Durable Research From Live Direction

Agents need both:

- stable context that rarely changes
- current context that changes often

Mixing them too aggressively causes memory rot.

### 3. Design For Multiple Agents Without Encoding Roles Too Early

The system should be usable by research agents and brainstorming agents, but without yet turning each agent into a fully specified persona.

So the memory should expose different types of material, not yet different full identities.

### 4. Help Generate New Ideas Without Losing Context

The goal is not merely recall.
The goal is contextual novelty.

The structure should help an agent ask:

- what is already known
- what is already attempted
- what themes recur
- what tensions remain open
- what adjacent idea would be genuinely new

### 5. Preserve Human Control

At this stage, the memory should remain human-curated.
Agents may read from it and reason with it, but they should not yet be allowed to rewrite the deeper structure on their own.

## Proposed Top-Level Structure

### `00 Navigation`

Purpose:
Entry layer for understanding the memory system quickly.

Planned note types:

- root index
- reading order
- map of memory layers
- how an agent should approach the system

### `01 Research Identity`

Purpose:
Capture the current shape of your research orientation without pretending it is final.

This should not be a rigid creed.
It should be a provisional representation of:

- recurring themes
- major interests
- persistent ambitions
- research temperament
- what kinds of problems matter to you

Why this exists:
Agents need a sense of your direction even before a formal philosophy is complete.

### `02 Research Terrain`

Purpose:
Represent the main subject landscape you are actually exploring.

This is where the strongest durable topics from `Code Archive 2` are translated into an agent-readable terrain map.

Likely contents:

- agentic systems
- tool intelligence
- research workflow
- knowledge architecture
- sovereignty and infrastructure
- strategic judgment

Why this exists:
An agent needs to know the territory before it can contribute usefully.

### `03 Active Investigations`

Purpose:
Represent what is currently alive.

This is where `LUX` contributes heavily.

Likely contents:

- current priorities
- active questions
- unfinished threads
- what should be revisited next
- what should not be forgotten right now

Why this exists:
Without this layer, an agent can understand the archive but still miss the present.

### `04 Project Memory`

Purpose:
Store the memory of important projects as reusable research objects.

Likely contents:

- project overviews
- why each project matters
- current status
- lessons extracted
- relation to wider research

Why this exists:
Projects are not just tasks.
They are embodiments of research priorities.

### `05 Idea Frontier`

Purpose:
This is the most important new folder.

It should track:

- open tensions
- adjacent possible ideas
- unexplored combinations
- ideas that are too similar to existing work
- next-step candidates with actual novelty

Why this exists:
This is the layer that can help agents generate useful new directions instead of recycling the archive.

### `06 Reference Compression`

Purpose:
Provide compact, high-signal documents that agents can read quickly.

This is the part most directly inspired by `LUX`.

Likely contents:

- compressed subject summaries
- operational briefs
- short environment snapshots
- first-read packets for agents

Why this exists:
Not every task should require reading long hub notes.

### `07 Memory Protocols`

Purpose:
Describe how the memory system works without yet imposing full behavioral law.

Likely contents:

- how files are updated
- difference between stable and living files
- reading order for agents
- curation rules
- anti-repetition logic

Why this exists:
The system needs explicit memory logic even before it gets explicit agent constitutions.

---

## Stable vs Living Files

This distinction is essential.

### Stable Files

These should change rarely.

Examples:

- terrain maps
- distilled research identities
- project lenses
- strategic principles

### Living Files

These should evolve regularly.

Examples:

- active investigations
- current priorities
- frontier questions
- near-term synthesis notes

The health of the whole system depends on not confusing the two.

---

## How This Surpasses Its Predecessors

### Beyond Code Archive 2

`Code Archive 2` is a strong curated archive, but it is still mainly knowledge architecture.

`Agent_S_Memory` goes further by asking:

- what should an agent read first
- what matters now
- what is unresolved
- what idea should emerge next

### Beyond LUX

`LUX` is operationally sharp, but strongly tied to specific active contexts and tool stacks.

`Agent_S_Memory` goes further by making the active context legible inside a larger long-term structure.

### New Capability

The new capability is not just memory.
It is guided research continuity.

That means an agent can move from:

- orientation
- to terrain
- to active context
- to frontier reasoning

without being trapped in either raw archive sprawl or overly narrow momentary summaries.

---

## Proposed Reading Order For Agents

An agent should eventually be able to read in this order:

1. `00 Navigation`
2. `01 Research Identity`
3. `03 Active Investigations`
4. `02 Research Terrain`
5. `05 Idea Frontier`
6. `04 Project Memory`
7. `06 Reference Compression`
8. `07 Memory Protocols`

This is intentionally not the same as a human browsing order.
For agents, active orientation should come earlier.

## First Prototype Outputs

The first creation pass should likely produce:

- a root index
- a research identity sketch
- a research terrain map
- an active investigations note
- a projects memory index
- an idea frontier note
- a compressed first-read packet
- a memory protocols note

These should be enough to create a functioning prototype without overcommitting too early.

---

## What Should Be Delayed To Phase Two

The following are excellent ideas, but should wait:

- fully specified agent roles
- autonomous memory updates
- strict behavioral instructions
- hard-coded voice constitutions
- detailed multi-agent operating laws

These belong after the structure proves itself.

---

## Current Recommendation

Proceed with `Agent_S_Memory` as a prototype memory environment rather than a final constitution.

That is the right level of ambition now.

It gives us:

- a strong organizing breakthrough
- room for future agent specialization
- better research continuity
- a place where innovation can arise from context

without locking you into a self-definition that is still evolving.
